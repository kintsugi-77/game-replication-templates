# 玩法实施卡 · 模板与交活 Prompt（通用）

> **用途**：游戏复刻实现阶段,每个里程碑填一张「玩法实施卡」,连同「交活 Prompt」发给执行 AI。
> 卡 = 这次做什么(导演填);Prompt = 按什么纪律做。真值数字统一放 `docs/<game>-spec.md`(见 spec 模板),卡**只引用、不抄数**。
> ★ 通用节奏:**先做最小可玩闭环(及格线),再逐系统铺开(评优线,把"最区别于同类的灵魂系统"重点做),最后打磨。**

## 模板 A · 玩法实施卡(两段式)

**导演填这 5 栏**(都拍得了板,不碰代码):
```
① 目标：这次能玩到核心循环的哪一段(玩家视角一句话)
② 及格判定：真实运行里"什么状态变了"算成功(可观察/可量化)
③ 真值/CONFIG：这次要对的数指向 spec 哪节;没定的标"占位/待拍板"
④ 验收点：对标原作我会盯哪几处
⑤ 不做/边界：这次切片不碰什么(防越界)
```
**执行 AI 在"动手前先做"回填、导演审这 5 栏**(导演填不了也不该填):
```
内置优先判断(哪些只改声明/配置 / 哪些写逻辑)
要用的 对象·组件(+ 声明出处)
要用的运行时 API(+ 引擎 API 类型定义/文档 出处)
参考哪个示例
状态机 phase 设计
```

## 模板 B · 交活 Prompt(把卡贴进【这次要做的】)
```
你用 AI 编码工具在本地工程 <path> 手写这个游戏的玩法。严格守以下工程纪律：

【纪律】
- 声明/逻辑分离：声明文件定义"有什么",逻辑文件定义"行为"。
- 内置优先：每个能力先问"能不能用引擎内置组件/配置实现";能 → 只改声明,不写逻辑。
- 新对象/组件先查字段(声明/API 文档)。
- 逻辑文件照骨架范例的固定段式写;运行时 API 先查引擎 API 类型定义/文档,不猜、不靠编译报错试探。
- 按玩法参考同类示例工程。
- 真值钉进 CONFIG 段,来源见 docs/<game>-spec.md;硬真值照搬,手感值标 [FEEL] 待导演调。
- 逻辑大了拆模块;状态机每帧 switch(phase),禁 setTimeout。
- 代码留 // WHY: 和 // GOTCHA:。
- 先在本地最小测试把核心逻辑/数值跑通,再部署到运行环境(逻辑先行,少走部署往返)。
- 真实运行环境验证过才算完——"编译过"≠"能玩"。

【这次要做的】
<把下面对应里程碑的实施卡贴在这里>

【动手前先做(等我确认再写代码)】
1. 复述你对这个切片核心循环的理解。
2. 回填技术 5 栏：内置优先 / 对象·组件 / API / 参考范例 / 状态机 phase,各标出处。
我确认后你再写;写完先离线验证,再部署 + 真实运行自测。
```

## 里程碑拆分指引(替代具体游戏的 M1…Mn)
- **M1 = 最小可玩闭环(及格线)**：核心循环跑通一轮,数值可先占位。
- **M2 … Mk = 逐系统铺开(评优线)**：每次接一个系统的真值并对拍原作;把**灵魂系统**(最区别于同类的)排在中段重点做。
- **M_last = 经营/体验打磨**：UI/HUD、存档、音效、美术替换占位。
- 每个里程碑用模板 A 填一张卡。


## 附：两张小抄(帮你审执行 AI 回填的技术栏)
**逻辑文件固定段式**：<按你的引擎逻辑文件结构填,如 类型别名 → 注册 → 配置(CONFIG) → 主入口 → 状态 → handlers>。

**内置优先怎么判**：
- 能力/表现类(行走、相机跟随、拾取、碰撞、物理)→ 先试纯引擎内置组件/配置(只改声明)。
- 游戏规则类(状态机、经济、数值系统)→ 写逻辑。
- 拿不准 → 让执行 AI 在"动手前先做"说明判断和出处,你再放行。
